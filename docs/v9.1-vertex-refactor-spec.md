**Status:** queued. Implement in Claude Code after the v9 → ~/code/live-factcheck-sidebar/ migration is complete.

**Goal:** Move all three model providers (Anthropic, Google Gemini, xAI Grok) onto Vertex AI's OpenAI-compatible chat-completions endpoint, behind a single GCP auth. Replace OpenAI (which is not available in Vertex AI for closed flagship models — only `gpt-oss` open-weights variants are there) with Grok 4.1 Fast as the third consensus voice. ## Architectural decision

**Single endpoint, single auth.** All API calls in `background.js` route through:

```
POST https://aiplatform.googleapis.com/v1beta1/projects/{PROJECT_ID}/locations/global/endpoints/openapi/chat/completions
```

with model names matching the actual Vertex AI Model Garden lineup as of 2026-05-19 (confirmed via the Agent Platform UI screenshot — current Anthropic shipping models on Vertex AI are Haiku 4.5 / Sonnet 4.5 / Opus 4.1; **Opus is explicitly out of scope for v9.1 per David's directive**, so we use only Sonnet 4.5 + Haiku 4.5):

## Auth approach

**Two tracks supported in v9.1**, picked via an Options dropdown:

### Track A — Service account JSON (default for solo dev / contest demo)

For David's personal testing and the initial contest submission. Simple and fast.

- David generates a service account in GCP Console: IAM → Service Accounts → Create → grant the **Vertex AI User** role on the `live-factcheck-sidebar` project → Create key → JSON, download.
- In the extension's Options, the user pastes the entire JSON into a textarea field.
- The extension stores it in `chrome.storage.local` (scoped, not page-readable).
- For each API call, `background.js` signs a JWT against the service account, exchanges it for an access token at the OAuth2 token endpoint, caches the token until 60s before expiry, then uses it in the `Authorization: Bearer` header for Vertex AI calls.

JWT signing in the extension service worker: use the Web Crypto API (`crypto.subtle.sign` with `RS256`). The service account JSON contains the private key; parse it with a minimal PEM-to-CryptoKey helper.

Risks: a service account JSON is a long-lived credential. Fine for solo / contest use; **not** acceptable for Chrome Web Store distribution.

### Track B — `chrome.identity` OAuth (production path)

For Chrome Web Store distribution.

- Pre-requisite (one-time, done by David in GCP Console): APIs & Services → Credentials → Create OAuth client ID → Type: Chrome Extension → enter the extension's Chrome Web Store ID. Get back the OAuth client ID. Add to `manifest.json` as:

  ```json
  "oauth2": {
    "client_id": "<the client id>.apps.googleusercontent.com",
    "scopes": ["https://www.googleapis.com/auth/cloud-platform"]
  },
  "permissions": ["storage", "alarms", "identity"]
  ```

- In Options, "Sign in with Google" button calls `chrome.identity.getAuthToken({interactive: true})`. Token is cached by Chrome; auto-refreshes.
- User also needs to enter their GCP **project ID** (free text field) — this is the project where the API calls bill against. Chrome can't infer this; user must specify.

Implementation tip: ship Track A first because it's testable on unpacked extensions immediately. Wire up Track B as the toggle option but mark it "Beta — requires Chrome Web Store install" in the UI until the extension is published.

### `background.js`

Replace the three current call functions with one unified function plus auth helpers:

```javascript
// New: unified Vertex AI caller. Replaces callAnthropic, callOpenAI, callGemini.
async function callVertexAI({ settings, publisher, model, system, user, maxTokens, temperature }) {
  const token = await getVertexAccessToken(settings); // see below
  const url = `https://aiplatform.googleapis.com/v1beta1/projects/${settings.gcpProjectId}/locations/global/endpoints/openapi/chat/completions`;
  const body = {
    model: `${publisher}/${model}`,
    messages: [
      ...(system ? [{ role: 'system', content: system }] : []),
      { role: 'user', content: user },
    ],
    max_tokens: maxTokens ?? 220,
    temperature: temperature ?? 0.4,
  };
  const r = await fetch(url, {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${token}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify(body),
  });
  if (!r.ok) {
    const errText = await r.text();
    throw new Error(`Vertex AI ${r.status}: ${errText.slice(0, 400)}`);
  }
  const data = await r.json();
  return data.choices?.[0]?.message?.content ?? '';
}

// Token acquisition — branches on auth mode.
async function getVertexAccessToken(settings) {
  if (settings.authMode === 'oauth') {
    return new Promise((resolve, reject) => {
      chrome.identity.getAuthToken({ interactive: false }, (token) => {
        if (chrome.runtime.lastError) return reject(chrome.runtime.lastError);
        resolve(token);
      });
    });
  }
  // Track A: service account JSON → JWT → access token, cached.
  return getServiceAccountToken(settings);
}

async function getServiceAccountToken(settings) {
  const cached = await chrome.storage.local.get(['vertexAccessToken', 'vertexTokenExpiresAt']);
  if (cached.vertexAccessToken && cached.vertexTokenExpiresAt > Date.now() + 60000) {
    return cached.vertexAccessToken;
  }
  // Sign JWT, exchange at oauth2.googleapis.com/token, cache result.
  // See https://developers.google.com/identity/protocols/oauth2/service-account#authorizingrequests
  // Minimum scope: https://www.googleapis.com/auth/cloud-platform
  const sa = JSON.parse(settings.serviceAccountJson);
  const jwt = await signServiceAccountJWT(sa); // implement using crypto.subtle + RS256
  const tokenResp = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      grant_type: 'urn:ietf:params:oauth:grant-type:jwt-bearer',
      assertion: jwt,
    }),
  });
  const tokenData = await tokenResp.json();
  if (!tokenData.access_token) throw new Error(`Token exchange failed: ${JSON.stringify(tokenData)}`);
  await chrome.storage.local.set({
    vertexAccessToken: tokenData.access_token,
    vertexTokenExpiresAt: Date.now() + (tokenData.expires_in - 60) * 1000,
  });
  return tokenData.access_token;
}
```

End-user setup guide:

1. Create a GCP project (or use an existing one). Note the project ID.
2. Enable the Vertex AI API on that project.
3. Pick auth mode:

## Open questions to resolve at implementation time

- **Regional vs global endpoint.** `aiplatform.googleapis.com/...locations/global/...` works for some partner models but not all. If a 404 surfaces for any provider, switch to a regional endpoint (`us-east5-aiplatform.googleapis.com/...locations/us-east5/...`). Confirm against current docs.
- **Quota.** Default Vertex AI quota for partner models may be too low for the consensus fan-out (3 parallel calls per conf-4/5 card). If `RESOURCE_EXHAUSTED` errors appear during testing, request a quota bump in GCP Console.
- **Streaming vs non-streaming.** Current extension is non-streaming. Keep that — streaming complicates the chitchat gate logic and offers no UX benefit for short fact-flag cards.
- **Backward compatibility.** Should v9.1 ship a one-time migration that reads old `apiKey` / `openaiKey` / `geminiKey` fields from storage and prompts the user to switch? Probably not — v9.1 is breaking; just show a clear "set up Vertex AI" empty state on Options first load.
